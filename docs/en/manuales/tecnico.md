# Technical manual

Detail of how the installation works under the hood. For normal use you don't need this;
it's useful for debugging, automating or auditing.

## Architecture

`agy` is a **native executable compiled in Go**, distributed as a "flat" binary (it's not
an npm package or a script). Design advantages:

- **Near-instant startup** of the TUI.
- **Low memory usage** with multiple background agents.
- **Security isolation** at the operating-system level to run AI-autogenerated commands
  (e.g. `sandbox-exec` on macOS).

## Official installer flow

The installer (`install.sh` / `install.ps1` / `install.cmd`) does, in order:

1. **Existence pre-check** — if `agy` already exists, it warns and exits (idempotent).
2. **Platform detection** — `uname -s` (OS), `uname -m` (arch: `amd64`/`arm64`) and
   **musl** detection on Linux (`/lib/libc.musl-*` or `ldd`). It builds an identifier
   like `linux_amd64`, `linux_arm64_musl`, `darwin_arm64`, etc.
3. **Manifest query** — downloads `…/manifests/<platform>.json`, which contains
   `version`, `url` and `sha512` of the binary for that platform.
4. **Download + verification** — pulls the payload to a *staging* directory and compares its
   **SHA512** against the manifest. If it doesn't match → **"Security Halt"** and aborts.
5. **Installation** — extracts/copies the binary to `~/.local/bin/agy` (or
   `%LOCALAPPDATA%\agy\bin\agy.exe`), applies `chmod +x` and on macOS removes quarantine
   (`xattr -d com.apple.quarantine`).
6. **Final setup** — invokes `agy install` so the binary itself completes its
   configuration.

## Directories and paths

| Element | Linux / macOS | Windows |
|---|---|---|
| Binary | `~/.local/bin/agy` | `%LOCALAPPDATA%\agy\bin\agy.exe` |
| Staging | under `$TMPDIR` / `%LOCALAPPDATA%\antigravity\staging` | same |
| Config / skills | `~/.gemini/` | `%USERPROFILE%\.gemini\` |
| Credentials | Keychain / Secret Service | Credential Manager |

## What the official one does NOT do: the PATH

The official installer **does not modify your `PATH`**. That's why this repo's wrappers add
that single piece. The correct pattern and why it matters:

```bash
# We gate on whether the LINE is already in the profile, NOT on the session's $PATH.
if ! grep -qsF 'export PATH="$HOME/.local/bin:$PATH"' "$PROFILE"; then
    printf '\n%s\n' 'export PATH="$HOME/.local/bin:$PATH"' >> "$PROFILE"
fi
export PATH="$HOME/.local/bin:$PATH"   # only affects THIS subshell
```

!!! danger "The classic PATH bug"
    Gating persistence with `if [[ ":$PATH:" != *":$HOME/.local/bin:"* ]]` decides whether
    to write to `.bashrc` based on the **current session's** PATH. If the directory is
    already on this session's PATH (but not in `.bashrc`), it **skips persistence** and
    `agy` disappears in the next terminal. The right question is not "is it on my PATH
    now?" but "will it be in my future terminals?" → gate with `grep` over the profile
    file.

!!! note "The `export` doesn't propagate to the parent"
    A child process cannot change the environment of the process that launched it. That's
    why, after running the installer, you need a new terminal or `source ~/.bashrc`.

## Install in a custom directory

```bash
curl -fsSL https://antigravity.google/cli/install.sh | bash -s -- --dir /opt/agy/bin
```

## Binary verification

```bash
file ~/.local/bin/agy          # native ELF/Mach-O
~/.local/bin/agy --version     # installed version
```
